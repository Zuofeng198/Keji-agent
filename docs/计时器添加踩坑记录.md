# 计时器添加踩坑记录

## 需求
在 AI 思考过程面板（thinking-panel）旁边加一个实时计时器，显示处理耗时。

## 错误历程

### 错误 1：只改了一个代码路径

**现象**：计时器没显示，无 JS 报错。

**根因**：前端有两个独立的流处理函数，各有各的思考面板：
- `sendPlanExecute()` — 生成计划时（plan 阶段）
- `sendChat()` — 主对话聊天时（最常用的路径！）

我只在 `sendPlanExecute()` 加了计时器，用户平时聊天走的是 `sendChat()`，完全没动到。

**教训**：改前端显示时，先梳理所有用到同一 UI 组件的代码路径，不能只改一个。

### 错误 2：用 `document.getElementById` 查未挂载的元素

**现象**：计时器没显示，无 JS 报错。

**根因**（`chat.js:37`）：
```javascript
hdr.innerHTML = '...<span id="planTimer">...</span>...';
var planTimerEl = document.getElementById('planTimer');  // ← null！
// ...
msgs.appendChild(thinkingPanel);  // ← 真正挂到 DOM 在第 56 行
```

`document.getElementById` 只在**已挂载到文档树**的元素中查找。而 `hdr` 刚通过 `createElement` + `innerHTML` 创建，还没 `appendChild` 到 `msgs` 上。所以返回 `null`，定时器回调里 `if (!planTimerEl) return;` 直接跳过，永不更新。

**修复**：用 `hdr.querySelector('.tp-timer')` 替代 `document.getElementById`。`Element.querySelector` 在元素的**子树**内查找，不需要元素已挂载到文档。

```javascript
// ✅ 正确：在已创建的元素子树内查找
var planTimerEl = hdr.querySelector('.tp-timer');
// ❌ 错误：文档中找不到未挂载的元素
var planTimerEl = document.getElementById('planTimer');
```

### 错误 3：定时器引用的 DOM 元素被替换后失效

**现象**：计划阶段的计时器在计划生成后（plan 事件）冻结显示 `00:00`。

**原因**：在 'plan' 事件处理中：
```javascript
hdr.innerHTML = '✅ 计划已就绪 <span class="tp-timer plan-done">...</span>';
```
这一行**整体替换了 `hdr` 的 innerHTML**，旧的 `planTimerEl` 引用的 DOM 节点被销毁了，但定时器还在用 `planTimerEl.textContent = ...` 更新它。因为我在 'plan' 事件中同时 `clearInterval(thinkingTimer)`，所以没造成报错，但计时器停止时的值可能是 `00:00`（如果计划生成很快）。

**教训**：定时器里引用的 DOM 元素可能被后续操作销毁，要注意生命周期。

## 总结

| 错误 | 根因 | 修复 |
|------|------|------|
| 只改了一个路径 | 不知道有 `sendChat()` 也在用思考面板 | 搜索所有用到同一 UI 的代码路径 |
| `getElementById` 返回 null | 元素未挂载到文档树 | 用父元素的 `querySelector` 替代 |
| 定时器引用失效 | innerHTML 替换销毁了旧元素 | 在销毁点 clearInterval |

## 后续改进

### 改进 1：智能滚动（不强制下拉）

**问题**：AI 思考/执行工具时，每次更新内容都 `msgs.scrollTop = msgs.scrollHeight`，用户无法往上翻看历史。

**修复**：添加 `_autoScroll()` 函数（`core.js`），只在用户接近底部时才滚动：

```javascript
function _autoScroll(el, threshold) {
  if (!el) return;
  threshold = threshold || 120;
  if (el.scrollTop + el.clientHeight >= el.scrollHeight - threshold) {
    el.scrollTop = el.scrollHeight;
  }
}
```

然后全局替换所有 `msgs.scrollTop = msgs.scrollHeight` 为 `_autoScroll(msgs)`：
- `sendPlanExecute()` 2 处
- `executePlanApproved()` 5 处
- `sendChat()` 6 处
- `addMessage()` / `addPhase()` 各 1 处

### 改进 2：工具执行状态可视化

**问题**：AI 执行工具时，`sendChat()` 的 tool_call 事件把 phaseDiv remove 掉了但没有创建新的状态标记，用户看到界面没变化以为卡住了。

**修复**：
- tool_call → `addPhase('⚡ 运行: 工具名', 'thinking')` 显示运行状态
- tool_result → `addPhase('✅ 工具名 完成', 'result')` + `setTimeout(remove, 1500)` 短暂显示后消失

### 改进 3：`executePlanApproved` 同一问题

该函数的 tool_call 同样没有 phase badge，补上同款。

## 本次新增的两大错误

### 错误 A：误判后端 SSE 事件名

**现象**：在 `sendChat()` 里加 `case 'tool_call'` / `case 'tool_result'` 的 phase badge，但永远不显示。

**根因**：后端 `/chat/stream` 路径的 SSE 只发 3 种事件：
- `think_token` — AI 思考文本（工具调用信息也嵌在文本里，如 `"🔧 调用工具: xxx"`）
- `answer` — 最终回答 token
- `done` — 流结束

**从来没有 `tool_call` / `tool_result` 事件**。这些 case 在 `sendChat()` 里存在了很久但全是死代码。添加的 phase badge 自然永远不会触发。

对比之下 `executePlanApproved` 函数（发到 `/chat/execute`）本来可能有这些事件，但 `/chat/execute` 这个路由后端根本没实现，所以也是死的。

**教训**：改前端前要先确认后端实际发了什么事件。不能看到 switch case 里有人写了 `tool_call` 就觉得这个事件真的会来。老老实实去读后端的 `adapter.py` 里的 `chat_stream()` 和 `SSEHook`。

### 错误 B：`_autoScroll` 跨文件引用导致崩溃

**现象**：`Uncaught ReferenceError: _autoScroll is not defined`，整个对话全白，消息消失。

**根因**：`_autoScroll` 定义在 `core.js`，被 `chat.js` 和 `history.js` 调用。但 WebView 缓存了旧版 `core.js`（虽然加了 `?v=2`），新 `chat.js` 加载时 `_autoScroll` 还不存在，`addMessage()` 直接崩溃。

**教训**：
1. 跨文件共享函数 → 一定要在被引用之前**硬编码一份内联定义**，防止单文件缓存更新不同步导致全局炸裂
2. `addMessage` 这种高频基础函数绝不能调用可能 undefined 的东西
3. 在 pywebview 环境里，文件级缓存不可控，cache busting（`?v=N`）不总能生效

**修复**：把 `_autoScroll()` 直接定义在 `chat.js` 顶部，不再依赖 `core.js` 的符号。

## 2026-05-12 批量踩坑：快捷命令栏 / compact / 弹窗

### 错误 1：CSS 忘记加版本号缓存穿透

**现象**：修改了 `style.css` 但浏览器一直显示旧样式，按钮还是方角的。

**根因**：
```html
<!-- 有 v=N：会更新 -->
<script src="/static/js/chat.js?v=7"></script>
<!-- 没 v=N：永不更新 -->
<link rel="stylesheet" href="/static/css/style.css">
```

JS 文件全部加了 `?v=N`，但 CSS 文件没加，浏览器一直用缓存的旧 CSS。

**教训**：**所有静态资源都要加版本号**，CSS 比 JS 更容易被缓存，因为浏览器通常先加载 CSS 再渲染。

**修复**：`<link rel="stylesheet" href="/static/css/style.css?v=7">`

### 错误 2：外部 CSS 缓存不可控时，依赖内联样式兜底

**现象**：即使加 `?v=N`，在 pywebview 环境里缓存仍然可能不更新。

**根因**：pywebview 的 WebView 组件缓存策略更激进，`?v=N` 不一定穿透。

**修复**：关键样式同时内嵌在 HTML 的 `<style>` 标签里，不依赖外部 CSS：
```html
<style>
.qc-toggle{...border-radius:20px...}
.qc-btn{...border-radius:14px...}
</style>
```
同时加 `<meta http-equiv="Cache-Control" content="no-cache">` 禁用页面缓存。

### 错误 3：POST 接口走了 GET

**现象**：`/compact` 报错"错误: 缺少 session_id"，实际上根本没请求到后端。

**根因**：`/api/compact` 是 `@router.post("/compact")`，但前端 `fetchCommand()` 通用函数发的是 `GET /api/command/compact`。两重错误：
- 路径不对（`/api/command/compact` vs `/api/compact`）
- 方法不对（GET vs POST）

**教训**：有特殊请求方法/参数的接口（如 `/compact` 需要 POST + body），不能走通用 `fetchCommand()`，必须单独处理。

**修复**：前后端都单独处理 `/compact`，不经过通用 GET 命令通道。

### 错误 4：命令执行后没有 loading 状态

**现象**：点 `/compact` 后界面没反应，用户不知道在运行。

**根因**：调 LLM 生成摘要需要几秒到十几秒，但前端直接 await，没有任何中间状态。

**教训**：所有耗时操作（调 LLM、网络请求）都要有一个"执行中..."的视觉反馈。

**修复**：发请求前添加 `phase-badge` 状态，完成/失败后移除。

### 错误 5：用浏览器原生 confirm 而不是自定义弹窗

**现象**：`/new` 和 `/clear` 弹窗是浏览器自带的，和 UI 风格不一致。

**根因**：直接用 `confirm()`，不知道代码库里已有 `showConfirm()` / `showDangerConfirm()`。

**教训**：改代码前先搜一下有没有现成的 UI 组件可以用。`core.js` 里已经有了完整的模态框系统。

**修复**：用 `showConfirm()` / `showDangerConfirm()` 替代所有 `confirm()`。

### 小结

| 错误 | 类型 | 教训 |
|------|------|------|
| CSS 没加 `?v=N` | 缓存 | 所有静态资源都要版本号 |
| pywebview 缓存不可控 | 缓存 | 关键样式内联到 HTML |
| POST 接口走 GET | 接口对接 | 特殊请求单独处理，不走通用通道 |
| 无 loading 状态 | UX | 所有耗时操作都要有中间反馈 |
| 用原生 confirm | 代码复用 | 先搜代码库，别重复造轮子 |
